# meta-archives.xyz — Project Specification

## 1. 프로젝트 개요

**meta-archives.xyz**는 디지털 아카이브를 수집·보존하는 플랫폼이 아니라,  
이미 존재하는 디지털 아카이브들을 **비평의 대상으로 삼는 메타-아카이브**이다.

이 프로젝트의 핵심 목적은 다음과 같다.

- 디지털 아카이브가 사용하는  
  메타데이터, 기술 구조, 분류 방식, 접근 정책이  
  어떤 세계관과 기억 방식을 전제하고 있는지를 드러낸다.
- 각 아카이브에 대한 **비평 글 자체를 기록물로 남긴다**.
- 비평을 데이터로, 데이터를 다시 비평으로 전환하는 실험을 수행한다.

---

## 2. 현재 단계와 1차 목표

### 현재 단계
- 메타데이터 스키마 설계 완료
- 개별 아카이브 단위의 메타데이터 기술 및 비평 작성 시작
- 작성자는 **콘텐츠(메타데이터 + 비평글)에 집중**
- 구현/빌딩은 **OpenCode가 전담**

### 1차 목표
- 각 아카이브마다 **HTML 기반의 상세 페이지 1개를 완성**
- 전체 사이트는 **정적 사이트(static site)** 로 구축
- 빌드 결과는 meta-archives.xyz에 배포

---

## 3. 역할 분담

### 작성자(콘텐츠 담당)
- 각 아카이브에 대해:
  - 메타데이터 작성
  - 비평 글(Markdown) 작성
- 스키마 준수
- Git 저장소에 콘텐츠 파일 추가

### OpenCode(구현 담당)
- 전체 웹사이트 빌딩 구조 설계 및 구현
- 콘텐츠 파일을 HTML 페이지로 변환
- 정적 사이트 생성 및 배포 자동화
- 콘텐츠 추가가 쉬운 구조 유지

---

## 4. 콘텐츠 구조 (Source of Truth)

각 아카이브는 **파일 2개**로 구성한다.


### 4.1 메타데이터 (YAML)

- 구조화된 데이터
- 스키마 기반
- 값이 없을 경우 `null` 또는 `unknown` 사용
- 관찰 시점 기반 필드 포함 (status, http_code 등)

### 4.2 비평 글 (Markdown)

- 자유 서술
- 최소 권장 섹션:
  - 1차 기술 (웹이 스스로 말하는 것)
  - 비평 (왜 이 아카이브가 수집 대상인가)
  - 관찰 메모

---

## 5. 웹사이트 필수 페이지 구성

### 5.1 목록 페이지 `/`

- 전체 아카이브 리스트
- 표시 항목:
  - title
  - year_started
  - status
  - tags
  - short description
- 기능:
  - 필터: status, tag, language
  - 정렬: 최신 등록순(기본), 이름순

### 5.2 상세 페이지 `/archives/<slug>/`

- 상단
  - title
  - 원본 URL
  - status + http_code
  - observed_at
- 본문
  - 메타데이터 테이블 (YAML 기반)
  - 비평 글(Markdown 렌더)
- 하단
  - 업데이트/관찰 날짜
  - (선택) GitHub 링크

### 5.3 About 페이지 `/about/`

- 프로젝트 미션
- 메타데이터 스키마 설명
- "아카이브를 비평하는 아카이브"라는 선언

---

## 6. 빌드 및 구현 요구사항 (OpenCode)

### 6.1 기술적 요구

- 정적 사이트 생성 방식
- HTML / CSS / 최소한의 JS
- 프레임워크 선택 자유 (예: Eleventy, Astro, custom build)
- 결과물은 순수 HTML로 배포 가능해야 함

### 6.2 빌드 규칙

- `data/archives/*.yml` + `content/archives/*.md` 결합
- slug는 파일명 또는 frontmatter 기준으로 단일화
- 필수 필드 누락 시 빌드 에러 또는 경고

### 6.3 자동화(권장)

- GitHub Actions 또는 동등한 CI
- 커밋 → 빌드 → 배포 자동화
- URL/slug 중복 검사
- YAML 문법 검증

---

## 7. 설계 원칙 (중요)

- 이 프로젝트는 CMS가 아니다.
- 실시간 수정/로그인은 필요 없다.
- 데이터와 콘텐츠는 Git 히스토리를 통해 시간성을 가진다.
- "완결된 아카이브"가 아니라  
  **관찰 중인 아카이브들의 집합**임을 전제로 한다.

---

## 8. 장기 확장 (현재는 구현하지 않아도 됨)

- 메타데이터 기반 비교 뷰
- 상태 변화(history) 시각화
- 스키마 버전 관리
- CSV/JSON export
- 연구용 데이터셋 공개

---

## 9. 요약

- 작성자는 **글을 쓰는 사람**
- OpenCode는 **글이 놓일 구조를 만드는 사람**
- meta-archives.xyz는  
  "아카이브를 저장하는 사이트"가 아니라  
  **아카이브를 읽고 기록하는 하나의 장치**다.
